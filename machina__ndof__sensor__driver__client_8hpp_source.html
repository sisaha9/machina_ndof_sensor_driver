50 /**
51  * @brief Class to handle the client side of the Machina NDof Sensor Driver
52  *
53  * This class provides an interface to the Machina NDof Sensor Driver to
54  * request sensor data and store it for further processing.
55  *
56  * The @ref MachinaNDofSensorDriverClientNode class spawns multiple instances
57  * of this class to handle multiple sensors.
58  *
59  * The class uses a client to request one sensor's data from the driver and
60  * stores it. The data is then sent to the
61  * @ref MachinaNDofSensorDriverClientNode via the @ref FillSensorOutput method.
62  *
63  * A thread is spawned to continuously request data from the driver at a
64  * specified interval. Since this is concurrent with the main thread, we
65  * have defined 2 modes of operation:
66  * - Lock-free using TACO from
67  * iceoryx (https://iceoryx.io/latest/advanced/iceoryx_hoofs/#concurrent)
68  * - Using shared_mutex
69  * You can toggle between these modes by setting -DENABLE_LOCKFREE to ON/OFF
70  * ON for lock-free and OFF for shared_mutex. More experiments and data needs
71  * to be collected but right now TACO enables a significant speedup but it
72  * blocks access to the data often at high frequencies. Thinking of swapping to
73  * a 2 element container in the future so that the producer can always write to
74  * the 2nd element. If the 2nd element is being written to (can use atomic to
75  * check) then the consumer can read from the 1st element. This way we can
76  * avoid blocking. This is a future update.
77  *
78  * A future update to be made here is to use TypeAdapters to reduce the
79  * number of memory copies. If we have 2 nodes in the same container and
80  * use type adapters, we could get the data, send it in the exact same format
81  * to the other node and it can use it directly. This would be a zero copy
82  *
83  * Additionally, unlike @ref MachinaNDofSensorDriverServiceNode, this class
84  * and @ref MachinaNDofSensorDriverClientNode have multiple callbacks sent to
85  * the ROS 2 executor. Hence, a simple StaticSingleThreadedExecutor (defined
86  * in the CMake of the @ref MachinaNDofSensorDriverServiceNode) won't work
87  * here if we want to hit high frequencies. We use a MultiThreadedExecutor
88  * (defined in the CMake) and set a new callback group here so that the
89  * executor runs at least the callbacks defined here in parallel with other
90  * callbacks on the executor.
91  */
